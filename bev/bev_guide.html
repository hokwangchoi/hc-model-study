BEV (Multi-Camera Tokenizer)

BEVFormer-style: learnable BEV queries cross-attend to 6 camera feeds through deformable attention, producing a dense top-down feature map. The perception backbone of modern autonomous driving.

The Core Problem: 6 Cameras → One Top-Down Map

A self-driving car has cameras mounted around the vehicle, each seeing a different slice of the surrounding world. The planning stack needs a single coherent top-down view — a feature map in the car's ego-centric frame showing everything nearby. The job of the BEV tokenizer is to turn multi-view imagery into that feature map.

Why this matters. Everything downstream — 3D object detection, lane segmentation, motion forecasting, trajectory planning — lives on this top-down feature map. A high-quality BEV tokenizer is the foundation of a modern AV stack. NVIDIA Drive, Tesla's occupancy network, Waymo's perception, and most academic self-driving research all use variants of this approach.

1. Two Approaches: LSS vs BEVFormer

There are two dominant families for building the BEV feature map. They solve the same problem in very different ways:

ApproachHow it builds BEV featuresProsCons
LSS (Lift-Splat-Shoot)
Philion & Fidler, 2020		 Predict per-pixel depth distribution. "Lift" each pixel to a frustum of 3D points weighted by depth probability. "Splat" into BEV cells. Simple, parallelizable, no iterative attention. Capped by how well the depth network can predict from a single image.
BEVFormer
Li et al., 2022 (this guide's focus)		 Learnable BEV queries cross-attend to camera features. Reference points from camera projection. Deformable attention keeps it tractable. No depth prediction needed. Geometry baked in via projection. Scales to more cameras. Complex attention mechanics. Needs custom CUDA kernel to be fast.

Both produce the same final output: a dense BEV feature map. The field has consolidated around BEVFormer-style for research benchmarks; production systems often combine ideas from both.

2. Full Model Pipeline

The full BEVFormer architecture from camera images to BEV feature map:

Three observations worth staring at:

  • The image features are static across the L BEVFormer layers — computed once by the backbone, then fed into every layer's cross-attention. Same for the camera geometry (reference points + validity masks).
  • The BEV queries shape-preserve through the entire stack: [B, 40000, d] in, [B, 40000, d] out after every layer. The last layer's output gets reshaped into a 2D grid [B, 200, 200, d] for the heads.
  • The only trainable inputs are the images and the BEV queries. Intrinsics and extrinsics are calibration constants — for a fixed rig, you can precompute all reference points once at startup.

3. Camera Projection: The Geometry Hook

The thing that makes BEV work is that every BEV cell has a known 3D location in the car's ego frame. That means we can project it to 2D pixel coordinates in every camera — without any neural network — just using linear algebra. This is how the cross-attention knows where to look in each image.

For a fixed camera rig, steps 1–3 are entirely deterministic — you only need intrinsics and extrinsics, no neural network. Production systems precompute the reference points once at startup and store them as a static lookup table. Dynamic cameras (e.g. pan-tilt-zoom) would recompute per frame.

4. Deformable Spatial Cross-Attention

With reference points in hand, each BEV query needs to sample features from the relevant pixels in each camera. Plain cross-attention from every query to every pixel is too expensive. Deformable attention solves this by only sampling K points per query per camera.

The full operation for each query q in BEV cell with ego position p_ego:

# Offsets and attention weights — predicted from the query
offsets = Linear(q).reshape(n_cam, K, 2)       # [6, 4, 2]
weights = softmax(Linear(q))                    # [6, 4]

# Reference point — computed outside, from camera projection
ref_uv = project(p_ego, K_intrinsic, T_ego_to_cam)   # [6, 2]

# Sample points
sample_pts = ref_uv.unsqueeze(1) + offsets       # [6, 4, 2]

# Bilinear-interpolate features from each camera at each sample point
sampled = grid_sample(img_feats, sample_pts)   # [6, 4, d]

# Weighted sum (masked so invalid cameras contribute 0)
q_new = (weights.unsqueeze(-1) * sampled).sum(dim=(0, 1))  # [d]

5. Code Walkthrough

project_ego_to_image — the camera projection function ▼

Purpose: Take the 3D position of every BEV cell (in ego frame) and project it through each camera's intrinsics and extrinsics to get a 2D pixel coordinate in that camera's image. Also returns a validity mask — a BEV cell 20m in front of the car is NOT visible to the rear-view camera.

def project_ego_to_image(points_3d, intrinsics, ego_to_cam, img_h, img_w):
    # points_3d:  [N_pts, 3]  — BEV cell centers in ego frame
    # intrinsics: [B, N_cam, 3, 3]
    # ego_to_cam: [B, N_cam, 4, 4]
    #
    # Returns:
    #   uv_norm [B, N_cam, N_pts, 2]  — normalized pixel coords ∈ [-1, 1]
    #   valid   [B, N_cam, N_pts]     — True iff cell visible to camera

    # Homogeneous 3D → apply extrinsics (ego → camera frame)
    ones = torch.ones(N_pts, 1)
    p_ego_h = torch.cat([points_3d, ones], dim=-1).T
    p_cam_h = ego_to_cam @ p_ego_h                  # [B, N_cam, 4, N_pts]

    # Apply intrinsics: K · p_cam, divide by Z
    pixels_h = intrinsics @ p_cam_h[..., :3, :]
    z = pixels_h[..., 2:3, :].clamp(min=1e-5)
    uv = pixels_h[..., :2, :] / z                 # pixel (u, v)

    # Validity: in front of camera AND within image bounds
    z_pos = p_cam_h[..., 2, :] > 0
    u, v = uv[..., 0, :], uv[..., 1, :]
    valid = z_pos & (u >= 0) & (u < img_w) & (v >= 0) & (v < img_h)

    # Normalize to [-1, 1] for grid_sample
    uv_norm = torch.stack([
        2.0 * u / (img_w - 1) - 1.0,
        2.0 * v / (img_h - 1) - 1.0,
    ], dim=-1)

    return uv_norm, valid

Key insight: this function doesn't train anything. It's pure linear algebra with learned camera calibration. For a fixed camera rig, the output depends only on the BEV grid (fixed) and calibration (fixed) — you can run it once at startup and cache the result as a lookup table.

SpatialCrossAttention — the deformable cross-attention module ▼

Purpose: For each BEV query, sample K points from each camera's feature map at reference_point + predicted_offset, then weight-sum the samples using predicted attention weights.

class SpatialCrossAttention(nn.Module):
    def __init__(self, d_model, n_cams=6, n_points=4, n_heads=8):
        # Per-query: predict (offset, weight) for each (cam, head, point)
        self.offset_proj = nn.Linear(d_model, n_cams * n_heads * n_points * 2)
        self.weight_proj = nn.Linear(d_model, n_cams * n_heads * n_points)
        self.value_proj = nn.Linear(d_model, d_model)
        self.output_proj = nn.Linear(d_model, d_model)

    def forward(self, queries, img_features, ref_points_norm, valid_mask):
        # queries:            [B, N_q, d]
        # img_features:       [B, N_cam, d, H_feat, W_feat]
        # ref_points_norm:    [B, N_cam, N_q, 2]
        # valid_mask:         [B, N_cam, N_q]

        values = self.value_proj(img_features)
        offsets = self.offset_proj(queries).reshape(
            B, N_q, N_cam, n_heads, n_points, 2)
        attn_logits = self.weight_proj(queries).masked_fill(
            ~valid_mask, float("-inf"))
        weights = F.softmax(attn_logits.reshape(B, N_q, -1), dim=-1)

        # For each camera, grid_sample values at (ref + offset)
        output = 0
        for c in range(N_cam):
            sample_pts = ref_points_norm[:, c].view(B, N_q, 1, 1, 2) \
                + offsets[:, :, c] * scale
            sampled = F.grid_sample(
                values[:, c], sample_pts.reshape(B, N_q * H * W, 1, 2),
                mode="bilinear", align_corners=True)
            # Weighted sum over heads × points
            output = output + (weights[:, :, c].unsqueeze(-1) * sampled).sum()

        return self.output_proj(output)

Production detail: the for c in range(N_cam) loop is bad — you want the whole thing fused into one kernel. The standard implementation is MSDeformAttn, a custom CUDA kernel that processes all cameras and queries in one launch. NVIDIA Drive ships a further-optimized variant with FP16 and Tensor Core utilization.

BEVFormer — the full model ▼

Purpose: Tie it all together — image backbone, learnable BEV queries, camera projection, transformer layers, output BEV feature map.

class BEVFormer(nn.Module):
    def __init__(self, bev_h=200, bev_w=200, d_model=256,
                 n_cams=6, n_layers=6, ...):
        self.backbone = ImageBackbone(out_channels=d_model)
        self.bev_queries = nn.Parameter(torch.zeros(bev_h * bev_w, d_model))
        self.bev_pos_embed = nn.Parameter(torch.zeros(bev_h * bev_w, d_model))
        # Precomputed 3D positions of each BEV cell in ego frame
        self.register_buffer("bev_points_3d",
                             make_bev_grid_points(bev_h, bev_w, 0.5, 0.0))
        self.layers = nn.ModuleList([
            BEVFormerLayer(d_model, n_heads=8, n_cams=n_cams, n_points=4)
            for _ in range(n_layers)
        ])

    def forward(self, images, intrinsics, ego_to_cam):
        B, N_cam, _, H, W = images.shape

        # 1. Per-camera backbone (shared weights, batched via reshape)
        feats = self.backbone(images.reshape(B * N_cam, 3, H, W))
        feats = feats.reshape(B, N_cam, self.d_model, self.feat_h, self.feat_w)

        # 2. Project BEV 3D points → 2D pixel refs in every camera
        ref_points, valid = project_ego_to_image(
            self.bev_points_3d, intrinsics, ego_to_cam, H, W)

        # 3. Initialize BEV queries
        bev_q = self.bev_queries.unsqueeze(0).expand(B, -1, -1)
        bev_q = bev_q + self.bev_pos_embed.unsqueeze(0)

        # 4. Transformer stack
        for layer in self.layers:
            bev_q = layer(bev_q, feats, ref_points, valid)

        # 5. Reshape to 2D feature map
        return bev_q.reshape(B, self.bev_h, self.bev_w, self.d_model)

7. Critical Optimizations

7.1 Deformable attention CUDA kernel ▼

The naive deformable attention in pure PyTorch loops over cameras and calls grid_sample each time. This is memory-bound and slow. The production kernel (MSDeformAttn from Deformable-DETR, adapted in MMDet3D and NVIDIA Drive):

  • Fuses the whole operation (offset prediction + grid_sample + weighted sum) into one CUDA kernel launch.
  • Each thread block handles a contiguous chunk of queries, with image features and offsets staged in shared memory.
  • On Ampere / Orin, FP16 + Tensor Cores give 2–3× speedup over the PyTorch path.

Without this kernel, BEVFormer is ~5× slower at inference. Any serious deployment installs it from source.

7.2 Shared backbone, batched cameras ▼

All 6 cameras share one CNN backbone's weights. Rather than calling it 6 times sequentially, stack the cameras into the batch dimension:

# Bad: 6 sequential calls
feats_per_cam = [backbone(images[:, c]) for c in range(6)]

# Good: one batched call
feats = backbone(images.reshape(B * 6, 3, H, W))
feats = feats.reshape(B, 6, d, H_feat, W_feat)

One GPU kernel, 6× batch size. Saves the cost of 5 launch overheads and lets the scheduler fully utilize Tensor Cores.

7.3 Precompute camera geometry ▼

For a fixed rig (intrinsics + extrinsics don't change), project_ego_to_image is deterministic. Precompute once at startup:

  • Reference points (2D pixel coords, normalized) — a [N_cam, N_bev, 2] tensor
  • Validity masks — [N_cam, N_bev] bool tensor

These become static inputs to the deformable attention kernel. For BEVFormer-base on nuScenes, this is ~2 MB of lookup tables. Saves ~1 ms per frame vs. recomputing.

For dynamic rigs (PTZ cameras, moving sensors), recompute per frame but keep the math on GPU.

7.4 Temporal self-attention (not in this implementation) ▼

The full BEVFormer has an extra layer: temporal self-attention that attends to the BEV feature map from the previous frame, warped by ego-motion. This:

  • Doubles perception quality for moving objects (velocity estimation, occlusion reasoning)
  • Only costs 1 extra attention per layer (one previous BEV tensor, carried over frames)
  • Analogous to Mamba's state-carrying: constant memory overhead, not growing

Almost every production BEV stack uses some form of temporal aggregation. StreamPETR takes this further with explicit object queries that persist across frames.

7.5 Quantization — mixed-precision is the sweet spot ▼
ComponentQuantizationWhy
Image backboneINT8 (PTQ)Mature, biggest compute savings
BEV queries + embeddingsFP16Small, always loaded; precision matters
Cross-attention projectionsINT8Works well
Deformable sample/sum kernelFP16grid_sample bilinear precision matters
HeadsFP16Keep precision near the output

Mixed INT8/FP16 gives ~2× throughput with <1% mAP drop on nuScenes benchmarks.

8. Deployment on Orin / Drive

Production targets for autonomous driving perception are strict: 10 Hz minimum for urban, 20 Hz for highway. For a 50 ms budget, where does the time go on Orin AGX?

StageRough budgetBottleneck
Image backbones (× 6)20–30 msCNN compute; Tensor Core-bound
Deformable cross-attention (× L)10–15 msMemory-bound scatter/gather
Self-attention over BEV5–10 msN_bev² — can blow up without care
FFN + norms5 msGEMMs, fine
Heads5–10 msDepends on what heads are attached
The self-attention N² trap. A 200×200 BEV grid has 40 000 queries. Full O(N²) self-attention has 1.6 billion scores per layer per head — that's ~13 GB of attention state at FP16 for a single layer with 8 heads. Blows past Orin's unified memory budget. Production systems use windowed self-attention (8×8 local windows) or replace self-attention with sparse / deformable variants to stay within bounds.

Knobs for tight budgets (ordered from biggest win to smallest):

  1. Reduce image resolution: 1600×928 → 800×464 saves ~4× backbone FLOPs
  2. Smaller BEV grid: 200×200 → 128×128 saves ~2.5× (N² term in self-attention)
  3. Skip temporal self-attention (saves ~1 ms / frame)
  4. Use a lighter backbone (ResNet-18 / MobileViT instead of ResNet-50)
  5. Drop to INT8 on backbone (another ~2× on the biggest cost)

9. Connection to the Roadmap Ahead

BEV is the perception substrate for several upcoming models:

  • VLA models (Pi0, OpenVLA) — a robot VLM that consumes multi-camera input will reuse a BEV-style front end. The BEV feature map becomes the visual context for language-conditioned action generation.
  • Occupancy networks — direct generalization: replace the 2D BEV grid with a 3D voxel grid. Same cross-attention mechanics, extra Z dimension. Tesla's FSD uses this.
  • Trajectory diffusion planners — the BEV feature map is the conditioning input for a diffusion-based planner predicting future ego trajectories. Returns us to the DiT backbone (you already saw diffusion!).
  • World models — BEV features at time t plus an action predict BEV features at time t+1. Mamba-like recurrent architectures fit this naturally.
