Mamba (Selective SSM)

A transformer alternative built on linear recurrence. No KV cache. Linear in sequence length. Two modes of computation from one model — parallel scan for training, recurrent for inference.

The Core Idea: A Linear Recurrence

Strip away everything else, and Mamba is built on this recurrence:

# For each time step t, for each input channel d, run an N-dim state update:
h_t = A · h_{t-1} + B_t · x_t         # [N]
y_t = C_t · h_t + D · x_t              # scalar
#
# Key point: the recurrence is LINEAR in h. That means it's an associative
# operation, which means we can parallelize it across t even though it
# looks sequential. A transformer's attention is also parallelizable but at
# the cost of O(L²) memory. Mamba is O(L · N) with N ≈ 16 — a huge
# reduction for long sequences.

This is a classic State Space Model (SSM) — think of it as a continuous-time linear system, discretized. Mamba's contribution is making A, B, C, Δ functions of the input (not fixed parameters), which gives the model a gating mechanism on par with attention.

Why this is a big deal. Transformers scale like O(L²) in attention. SSMs scale like O(L · N) with tiny N. A 130M Mamba at L=32K uses ~100 MB of state; a 130M transformer's KV cache alone is several GB. For long-context inference, the memory math is a different league.

1. SSM → Selective SSM — what Mamba adds

Plain SSMs (S4) use fixed A, B, C. Efficient, parallelizable, but bad at picking which tokens matter. Mamba makes them input-dependent:

Intuitively: when you read a paragraph and hit a word like "however", Δ_t might spike to say "reset state, new thought coming"; when you hit a filler word, Δ_t shrinks to say "keep going, don't let this perturb the state". Plain SSMs can't do this.

2. Two Modes of Computation

The same recurrence admits two very different implementations. This is the most important thing to internalize about Mamba:

This dual-mode computation is rare and powerful. Transformers are "all parallel, all the time" — even at inference they do parallel attention against the cache. RNNs are "all sequential". Mamba gets both: parallel efficiency at train time, recurrent efficiency at inference.

3. The Mamba Architecture

A Mamba language model is just a stack of identical Mamba layers wrapped around token embedding and an LM head. No positional encoding (the recurrence inside each layer is intrinsically positional), no attention at all, no cross-layer anything unusual — the novelty lives entirely inside one layer.

Two things to notice:

  • Shape is preserved through every layer: [B, L, d_model] in, [B, L, d_model] out. No downsampling, no upsampling, no cross-attention to another stream. Just L layers of the same thing, stacked.
  • No positional encoding is added anywhere in the pipeline. The MambaBlock's recurrence (inside each layer) builds position information into the state as it walks left-to-right through the sequence. A transformer needs RoPE or learned positional embeddings; Mamba gets positions for free.

Now zooming in on what happens inside one Mamba layer — specifically the MambaBlock, which is the expand → mix → scan → gate → contract pattern:

Component roles

  • in_proj: expands dimensionality and creates two streams. Standard Linear.
  • conv1d (depthwise): small K=4 filter across time. Adds local context that the scan alone doesn't capture well. Depthwise = per-channel, no channel mixing (that's in_proj's job).
  • Selective scan: the heart. Computes N-dim state updates for every (batch, position, channel). The ONE operation unique to Mamba.
  • z gate: SiLU(z) × scan_output — elementwise multiplicative gating. Lets the model modulate what passes through.
  • out_proj: contracts back to d_model. Standard Linear.

5. Hardware-Aware Kernel (The FlashAttention Trick)

The selective scan naively needs an intermediate tensor of shape [B, L, D_inner, N] — for Mamba-B at L=2048 that's 50M elements per layer per batch. Materializing this in HBM and reading it back repeatedly would make the memory bandwidth bill enormous.

The official selective_scan_cuda kernel does what FlashAttention did for attention:

Same philosophy as FlashAttention: keep the big intermediate tensor SRAM-resident, recompute during backward if needed. Without this kernel, Mamba is slower than a transformer. With it, Mamba is 2–3× faster.

6. Code Walkthrough

The four core classes from mamba.py.

selective_scan_reference — the core operation (sequential) ▼

Purpose: Run the selective SSM recurrence. This Python version loops over time steps for clarity; the real implementation is a hardware-aware CUDA kernel.

def selective_scan_reference(u, delta, A, B, C, D_skip):
    B_size, L, D_size = u.shape
    N = A.shape[-1]

    # Discretize: dA = exp(delta · A), dB = delta · B. Input-dependent because
    # delta, B, C all come from x (they're the "selective" part).
    deltaA = torch.exp(delta.unsqueeze(-1) * A)                       # [B, L, D, N]
    deltaB_u = delta.unsqueeze(-1) * B.unsqueeze(2) * u.unsqueeze(-1)  # [B, L, D, N]

    h = torch.zeros(B_size, D_size, N, device=u.device, dtype=u.dtype)
    ys = []
    for t in range(L):
        h = deltaA[:, t] * h + deltaB_u[:, t]                           # [B, D, N]
        y_t = torch.einsum("bdn,bn->bd", h, C[:, t])                  # [B, D]
        ys.append(y_t)
    y = torch.stack(ys, dim=1)                                          # [B, L, D]
    return y + u * D_skip                                             # per-channel skip

Key insight: the loop over t is what makes this O(L). In practice we replace this with a parallel scan using the fact that the recurrence (A, b) · (A', b') = (A · A', A · b' + b) is associative.

MambaBlock — the expand-scan-gate-contract cell ▼

Purpose: One complete Mamba layer. In-projection doubles dimensionality into (x_ssm, z), runs conv + SSM on x_ssm, gates with SiLU(z), contracts back.

class MambaBlock(nn.Module):
    def __init__(self, d_model, d_state=16, d_conv=4, expand=2):
        super().__init__()
        self.d_inner = expand * d_model

        self.in_proj = nn.Linear(d_model, 2 * self.d_inner, bias=False)

        # Depthwise 1D conv for local mixing before the SSM
        self.conv1d = nn.Conv1d(
            self.d_inner, self.d_inner,
            kernel_size=d_conv,
            groups=self.d_inner,          # depthwise — no cross-channel mix
            padding=d_conv - 1,
        )

        # x_proj produces Δ_rank, B [N], C [N] — ALL data-dependent
        self.x_proj = nn.Linear(self.d_inner, dt_rank + 2 * d_state, bias=False)
        self.dt_proj = nn.Linear(dt_rank, self.d_inner, bias=True)

        # A is parameterized as log|A| so A = -exp(A_log) is always negative
        # Negative A ensures the recurrence decays (stable).
        self.A_log = nn.Parameter(torch.log(torch.arange(1, d_state + 1).repeat(self.d_inner, 1)))
        self.D = nn.Parameter(torch.ones(self.d_inner))

        self.out_proj = nn.Linear(self.d_inner, d_model, bias=False)

    def forward(self, x):
        xz = self.in_proj(x)                              # [B, L, 2·d_inner]
        x_ssm, z = xz.chunk(2, dim=-1)                    # each [B, L, d_inner]

        # Depthwise conv + SiLU
        x_ssm = self.conv1d(x_ssm.transpose(1, 2))[..., :L].transpose(1, 2)
        x_ssm = F.silu(x_ssm)

        # Data-dependent Δ, B, C
        x_dbl = self.x_proj(x_ssm)
        dt, B, C = torch.split(x_dbl, [dt_rank, d_state, d_state], dim=-1)
        delta = F.softplus(self.dt_proj(dt))

        A = -torch.exp(self.A_log.float())                # [d_inner, d_state]

        # The scan — the only operation unique to Mamba
        y = selective_scan_reference(x_ssm, delta, A, B, C, self.D)

        # Gate + contract
        return self.out_proj(y * F.silu(z))

Deployment note: The in_proj, out_proj, and conv layers are standard. Only selective_scan_reference needs replacing with the CUDA kernel at inference time.

MambaBlock.step — recurrent inference mode ▼

Purpose: Process one token at a time, carry a fixed-size state. This is the mode used in autoregressive generation. Memory is O(d_inner · d_state) per layer per batch — constant in sequence length.

def step(self, x_t, h, conv_cache):
    # x_t: [B, d_model] — ONE token
    # h:   [B, d_inner, d_state] — carried state
    # conv_cache: [B, d_inner, d_conv] — last d_conv inputs for the depthwise conv

    xz = self.in_proj(x_t)
    x_ssm, z = xz.chunk(2, dim=-1)

    # Conv is a dot product across kernel positions now (no sequence loop)
    conv_cache = torch.roll(conv_cache, shifts=-1, dims=-1)
    conv_cache[..., -1] = x_ssm
    x_ssm = torch.einsum("bdk,dk->bd", conv_cache, self.conv1d.weight.squeeze(1))
    x_ssm = F.silu(x_ssm + self.conv1d.bias)

    # Δ, B, C for this single token
    x_dbl = self.x_proj(x_ssm)
    dt, B, C = torch.split(x_dbl, [dt_rank, d_state, d_state], dim=-1)
    delta = F.softplus(self.dt_proj(dt))

    A = -torch.exp(self.A_log.float())
    dA = torch.exp(delta.unsqueeze(-1) * A)                    # [B, d_inner, d_state]
    dB = delta.unsqueeze(-1) * B.unsqueeze(1)                  # [B, d_inner, d_state]

    # Update state — O(d_inner · d_state) compute, same for every token
    h = dA * h + dB * x_ssm.unsqueeze(-1)

    y = torch.einsum("bdn,bn->bd", h, C) + self.D * x_ssm
    y = y * F.silu(z)
    return self.out_proj(y), h, conv_cache

Interview moment: the per-token cost is fully independent of L. A transformer at L=100,000 does an attention over 100,000 cached K/V pairs per token. Mamba just updates a small state tensor. This is the headline performance story for long-context inference.

Mamba (full model) — stack of MambaLayers ▼

Purpose: Wrap token embedding, L Mamba layers (each with residual + RMSNorm), final norm, and LM head. Almost identical to a transformer's top-level structure — only the mixer changes.

class Mamba(nn.Module):
    def __init__(self, vocab_size, d_model=768, n_layer=24, ...):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, d_model)
        self.layers = nn.ModuleList([
            MambaLayer(d_model, ...) for _ in range(n_layer)
        ])
        self.norm_f = RMSNorm(d_model)
        self.lm_head = nn.Linear(d_model, vocab_size, bias=False)
        self.lm_head.weight = self.embedding.weight       # tied weights

    def forward(self, token_ids):
        x = self.embedding(token_ids)
        for layer in self.layers:
            x = layer(x)                                  # residual inside
        x = self.norm_f(x)
        return self.lm_head(x)

What's notably absent: no positional encoding. The scan is intrinsically positional because state is built up left-to-right. Positions are implicit.

8. Critical Optimizations

8.1 The hardware-aware selective scan kernel ▼

Without the custom kernel, Mamba is impractically slow. The kernel:

  • Fuses discretize + scan + output into one CUDA launch.
  • Keeps the [B, L, D_inner, N] intermediate in SRAM — never writes it to HBM.
  • Uses Blelloch parallel scan within SRAM tiles.
  • Recomputes the scan during backward pass (standard gradient checkpointing trade).

Exactly the FlashAttention pattern, applied to a different memory-dominant intermediate. Installed via pip install mamba-ssm; source at github.com/state-spaces/mamba.

8.2 Mamba-2 — structured state-space duality ▼

Mamba-2 (Dao & Gu, 2024) connects SSMs and attention through "Structured State-Space Duality" (SSD). The practical wins:

  • 2–8× faster than Mamba-1 — reuses well-optimized matmul kernels.
  • Simpler A structure (scalar per head), much easier to implement.
  • Larger state dimension (N = 64–128 vs 16) fits cleanly in the new kernel.
  • Works better with Tensor Cores because the inner loop is a block matmul.

If you're choosing between Mamba-1 and Mamba-2 for a new deployment, use Mamba-2.

8.3 Hybrid models — Transformer + Mamba ▼

Pure Mamba struggles with some attention-friendly tasks like exact recall and in-context learning. Pure transformer struggles with long contexts. The answer is hybrid architectures:

  • Jamba (AI21, 2024) — 7:1 Mamba:attention ratio. First production-scale hybrid LLM.
  • Samba, Zamba — variations of the same pattern.
  • NVIDIA's own research (Nemotron-H, etc.) has been exploring hybrid SSM-Transformer designs.

Deployment implication: serving infra needs to understand both block types. A request may pass through mixed blocks with different memory profiles; allocation and scheduling have to account for that. This is one reason the JD mentions "hybrid SSM-Transformer" explicitly.

8.4 No KV cache, but the state still needs to be managed ▼

While there's no KV cache, the per-request state tensor [B, d_inner, d_state] + conv cache [B, d_inner, d_conv] still needs allocation:

  • For Mamba-B: state is 2 · 1536 · 16 = 49,152 values per layer per batch ≈ 100 KB at FP16.
  • × 24 layers ≈ 2.4 MB per request.
  • Compare with transformer KV cache at L=16K: ~800 MB for the same model.

This is what makes Mamba attractive for serving many concurrent long-context requests — you can pack far more of them into the same GPU memory.

8.5 Quantization — similar story to transformers ▼

The Linear layers (in_proj, out_proj, x_proj) quantize well to INT4/INT8, same as transformers. The scan state is harder — it's where precision matters most because errors compound through the recurrence. Rules of thumb:

  • Weights: INT4 AWQ/GPTQ works fine for the projections.
  • Activations: FP16 for the scan path; INT8 is risky.
  • State h: Keep FP16 or higher. Quantizing state is an open research area.
  • A matrix: stored as log|A|, always float — tiny so no need to quantize.

10. On Edge Hardware

Why Mamba is a good fit for edge inference:

  • No KV cache blow-up. Orin Nano's 8 GB is much more usable at long L with Mamba than with a transformer of equivalent size.
  • Recurrent mode's constant per-step cost fits cleanly with small batch sizes and real-time requirements.
  • The scan kernel is the risk factor. The official CUDA kernel targets sm_80+ cleanly; Jetson Orin is sm_87 but integrations vary. Check the mamba_ssm release matrix before committing.
Real talk: if you need a production LLM on Jetson today, a quantized transformer (vLLM, llama.cpp, TRT Edge-LLM) is still the default. Mamba is promising and the long-context story is compelling, but the inference toolchain is less mature. For serving long-context chat, the hybrid approach (Jamba-style) is probably what you'd ship rather than pure Mamba.

11. Connection to the Roadmap Ahead

Mamba's recurrent structure maps naturally onto several models in the roadmap:

  • VLA models (Pi0 and related): action chunking over long horizons benefits from a recurrence with fixed state. Mamba-style blocks are being explored for robotic action experts.
  • World models: latent dynamics over long trajectories — the compressed state vector is a useful inductive bias.
  • Robot foundation models: learning from long demonstration sequences. Hybrid architectures with Mamba blocks in the mid-stack are showing up in recent work.
