Robot Foundation Model

GR00T N1-style: one shared VLM backbone + DiT action head driving many different robot embodiments (arms, humanoids, quadrupeds) through heterogeneous I/O routing and a dual-rate System 1 / System 2 split.

The Core Idea: One Model, Many Robots

A robot foundation model is a single pretrained network that can drive different robots with different morphologies, action spaces, and sensors. The "foundation" part means it's pretrained on massive cross-embodiment data and fine-tuned for specific robots rather than trained from scratch per-robot.

The key insight: most of what a robot needs to know is embodiment-agnostic — what objects look like, what "put the cup on the table" means, how physics works. Only the final step of "which specific motors do I command?" is embodiment-specific. Sharing 95%+ of the model across embodiments while keeping thin per-embodiment I/O projections is what makes cross-embodiment transfer possible, and it's what distinguishes a foundation model from a single-task policy.

1. The Dual-System Architecture

A human driving a car isn't constantly re-deciding strategy. "I'm going home" is a slow, high-level judgment (often called "System 2" thinking). Individual steering corrections are fast, reflexive adjustments ("System 1"). GR00T applies this split to robot control — decoupling scene understanding from motor commands by rate:

Concretely, in a 1-second chunk:

  • System 2 runs once (~10 Hz target rate). Processes all cameras, the language instruction, proprioception, and produces context tokens. Expensive — this is where most of the parameters live (Eagle-2 VLM, ~1.5B params in real GR00T).
  • System 1 runs many times (~120 Hz target rate). Small DiT-style transformer that cross-attends to the cached System 2 output and generates motor commands through flow-matching Euler integration. Cheap per call because it skips all the vision/language encoding.

Without the split, you'd re-run the entire VLM at every action step — ~10× more compute for no quality gain since scene understanding doesn't change that fast.

2. Full Model Pipeline

Now the full picture, with heterogeneous I/O routing visible:

Three things to notice:

  • System 2's output feeds System 1 via cross-attention — the coral dashed wire. Every System 1 block queries the cached VLM tokens but does not modify them.
  • Per-embodiment routing is thin. Only three small Linear layers change per embodiment: proprio_in, action_in, action_out. The massive VLM and DiT blocks are shared.
  • The flow time + embodiment summary drives AdaLN-Zero conditioning inside every System 1 block — same mechanism as the DiT image model we built earlier, now generating action chunks instead of images.

3. Heterogeneous I/O — How Multi-Embodiment Routing Works

Different robots have fundamentally different proprioception and action spaces. The architectural trick is one shared model + many tiny per-embodiment I/O projections, routed by name.

4. Training on Cross-Embodiment Data

Training batches are labeled with the embodiment they come from, and routed through the matching heads. Typical recipe:

# Pseudo-code for one training step
for epoch in range(n_epochs):
    for batch in dataloader:
        embodiment = batch["embodiment"]    # e.g. "h1" or "franka_arm"

        # Route proprio through the embodiment's encoder
        proprio_tok = model.proprio[embodiment](batch["proprio"])

        # Shared VLM path
        vlm_tokens = model.encode_system2(batch, embodiment)

        # Action training: encode actions through THIS embodiment's projection
        t = torch.rand(B)
        x_t = (1 - t) * eps + t * batch["action"]
        v_pred = model.system1(x_t, t, vlm_tokens, embodiment)   # routes internally
        loss = F.mse_loss(v_pred, batch["action"] - eps)
        loss.backward()

5. Code Walkthrough

HeterogeneousProprioEncoder — routing by embodiment name ▼

The simplest form of multi-embodiment routing — a dict of MLPs, each matched to one embodiment's proprio dim.

class HeterogeneousProprioEncoder(nn.Module):
    def __init__(self, embodiments, d_model):
        self.encoders = nn.ModuleDict({
            name: nn.Sequential(
                nn.Linear(spec["proprio_dim"], d_model),
                nn.GELU(),
                nn.Linear(d_model, d_model),
            )
            for name, spec in embodiments.items()
        })

    def forward(self, proprio, embodiment):
        # Route to the matching encoder at forward time
        return self.encoders[embodiment](proprio).unsqueeze(1)

Why ModuleDict and not just a list? We want to route by name, not index, because different batches may come from different embodiments. A nn.ModuleDict gives us O(1) lookup by the embodiment string, and PyTorch tracks all the parameters for autograd.

System1Block — DiT-style AdaLN-Zero conditioning ▼

Each System 1 block modulates self-attention and FFN using (flow time + embodiment summary) via AdaLN-Zero — the exact same mechanism as our DiT image diffusion model, now generating actions.

class System1Block(nn.Module):
    def __init__(self, d_action, d_vlm, d_cond, n_heads):
        # AdaLN produces 6 modulation scalars from conditioning vector
        self.mod = AdaLNModulation(d_action, d_cond)
        self.self_attn = nn.MultiheadAttention(d_action, n_heads, batch_first=True)
        self.kv_proj = nn.Linear(d_vlm, d_action, bias=False)
        self.cross_attn = nn.MultiheadAttention(d_action, n_heads, batch_first=True)
        self.mlp = nn.Sequential(
            nn.Linear(d_action, 4 * d_action), nn.GELU(),
            nn.Linear(4 * d_action, d_action))

    def forward(self, x, cond, vlm_tokens):
        shift_a, scale_a, gate_a, shift_m, scale_m, gate_m = self.mod(cond)

        # 1. Modulated self-attention, gated residual
        y = modulate(self.norm_sa(x), shift_a, scale_a)
        attn_out, _ = self.self_attn(y, y, y)
        x = x + gate_a.unsqueeze(1) * attn_out

        # 2. Cross-attention to VLM (unmodulated — VLM is fixed per chunk)
        kv = self.kv_proj(vlm_tokens)
        ca_out, _ = self.cross_attn(self.norm_ca(x), kv, kv)
        x = x + ca_out

        # 3. Modulated FFN, gated residual
        y = modulate(self.norm_mlp(x), shift_m, scale_m)
        x = x + gate_m.unsqueeze(1) * self.mlp(y)
        return x

Comparison with DiT image model: identical AdaLN-Zero pattern, just with different input shapes. DiT conditions on (timestep, class label) → flat image patches. Here we condition on (flow time, embodiment summary) → action chunks. The abstraction generalizes beautifully.

RobotFoundationModel.generate_action — inference with caching ▼

The full inference pipeline shows the rate asymmetry explicitly: System 2 runs once, System 1 integrates the ODE.

@torch.no_grad()
def generate_action(self, images, token_ids, proprio, embodiment, n_steps=10):
    B = images.size(0)
    action_dim = self.embodiments[embodiment]["action_dim"]

    # --- System 2 — ONCE per chunk (expensive) ---
    vlm_tokens = self.encode_system2(images, token_ids, proprio, embodiment)

    # --- System 1 — 10 Euler steps (fast each) ---
    x = torch.randn(B, self.horizon, action_dim, device=images.device)
    dt = 1.0 / n_steps
    for i in range(n_steps):
        t = torch.full((B,), i * dt, device=images.device)
        v = self.system1(x, t, vlm_tokens, embodiment)   # cross-attends to cached VLM
        x = x + v * dt                                   # Euler step

    return x   # [B, H, action_dim] — routed to the right embodiment dim

Why this matters: the VLM is 80% of the compute. Caching its output across Euler steps is the single biggest win. This is also what lets the rate asymmetry work — System 2 at 10 Hz means one encode every 100ms; System 1 at 120 Hz means one action step every 8ms.

7. Critical Optimizations

7.1 Cache System 2 across System 1 steps ▼

Foundational. Without this, each Euler step re-runs the VLM — ~10× more compute for identical output. Same pattern as VLA, just more critical here because System 2 is a larger fraction of parameters in production.

7.2 Share cross-attention K/V within a chunk ▼

Inside one chunk prediction, the VLM context is fixed, so the keys and values for cross-attention in System 1 don't change across Euler steps. Compute them once, reuse 10×:

# Precompute K, V from VLM tokens (inside each S1 block)
kv_cache = [block.kv_proj(vlm_tokens) for block in s1_blocks]

# Reuse across Euler steps
for i in range(10):
    for block, kv in zip(s1_blocks, kv_cache):
        x = block.self_attn_and_cross(x, kv)      # kv from cache

Saves ~30% of System 1 cost per chunk on real GR00T-size models.

7.3 Embodiment-aware batching ▼

Training on mixed-embodiment data naïvely — random samples across embodiments — forces the model to switch between ModuleDict entries every sample, causing kernel-launch overhead. Best practice:

  • Batch-by-embodiment: each batch is homogeneous (all Franka, all H1, etc.)
  • Rotate batches by embodiment: epoch = 4 mini-epochs, one per embodiment
  • Or use gradient accumulation across per-embodiment microbatches

Significant speedup on GPU; minimal impact on training dynamics.

7.4 Freeze System 2 during fine-tuning ▼

After pretraining on large cross-embodiment data, the VLM backbone can be frozen for most downstream fine-tuning. Only train System 1 + per-embodiment I/O heads. Benefits:

  • ~5× less training memory (no S2 gradients or optimizer state)
  • Preserves generalist VLM capabilities (language, novel objects)
  • Enables rapid adaptation to new embodiments: add I/O heads, train for a few hours
  • Allows INT8 VLM inference during fine-tuning (since no gradients needed)

GR00T's training recipe explicitly uses this pattern after pretraining.

7.5 Quantization: INT8 System 2, FP16 System 1 ▼

Same as VLA. Vision + language quantize well to INT8; action head flow matching needs FP16 precision (Euler steps compound errors). Mixed precision gives ~2× speedup with minimal quality loss.

9. Open Research Questions

This module intentionally ends with open problems — we're at the frontier here, not a solved science:

  1. Scaling laws for robotics. Does doubling data and params predictably improve robot policies the way it does for LLMs? Early results from GR00T and Pi0.5 suggest yes, but the exponents seem different. Still an open empirical question.
  2. Reward-free pretraining. LLMs pretrain on next-token prediction — infinite web text, no labels. Robots currently need action labels, which are expensive. Can world modeling or self-supervised video substitute?
  3. Sim-to-real transfer. Simulation data is essentially free but has a distribution gap to real robots. NVIDIA Isaac, Cosmos, and academic approaches aim to close this, but it's still an active pain point.
  4. Cross-morphology transfer. If I train on arms and quadrupeds, does a humanoid get zero-shot capability? Current evidence: yes for perception, partially for high-level skills, no for low-level balance control. Finding what transfers is a research area.
  5. Streaming chunks. Chunked prediction is efficient but can't react mid-chunk. Receding-horizon chunk prediction with continuous rollout is an active research direction.
  6. Real-time RL on foundation policies. Fine-tune a pretrained VLA/VLM with on-robot RL. PPO on flow-matched policies is mostly unexplored.

10. Closing the Roadmap

This is the last module. Looking back at the full journey:

  • Transformer — the attention mechanism that everything is built on
  • ViT — image patches as tokens, extending transformers to vision
  • VLM — fusing vision and language through cross-modal transformer blocks
  • Diffusion (DiT) — generative modeling with AdaLN-Zero conditioning
  • Mamba — an alternative to attention with constant-memory inference
  • BEV — multi-camera 3D perception with deformable cross-attention
  • VLA — adding action generation via flow matching on top of VLM
  • Robot Foundation Models — multi-embodiment, dual-system, heterogeneous I/O

Every module builds on the previous ones. The Robot Foundation Model genuinely reuses every earlier building block:

  • Transformer attention (System 2 blocks, System 1 self-attention)
  • ViT-style patchification (vision encoder)
  • VLM cross-modal fusion (VLM backbone)
  • DiT AdaLN-Zero conditioning (System 1 blocks)
  • Flow matching training + Euler inference (from VLA)
  • Action chunks (from VLA)
  • (BEV and Mamba are the "parallel tracks" — BEV could replace the vision encoder for multi-camera setups; Mamba could replace the transformer for longer context)

The pattern "chain together pretrained building blocks + small adapters" is how modern foundation models are assembled. This roadmap has built the building blocks. Robot FM is one assembly — others (e.g., multimodal reasoning agents, video generation models, code agents) follow the same playbook with different pieces.
