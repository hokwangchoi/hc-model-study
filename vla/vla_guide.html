VLA — Vision-Language-Action

Pi0-style: a VLM backbone (vision + language + proprioception) paired with a parallel action expert that generates continuous action chunks via flow matching. The architecture behind modern robot foundation models.

The Core Idea: Instruction In, Action Chunk Out

A VLA takes what the robot sees (cameras), what it's been asked to do (language), and where it currently is (proprioception), and outputs a sequence of future joint commands — an action chunk of H=50 timesteps, about one second of motion at 50 Hz.

Why chunks? Predicting one action per observation would require running the 3B-param VLA at 50Hz — infeasible on robot hardware. Predicting chunks decouples prediction rate (1–5Hz, expensive) from execution rate (50Hz, just step through the buffer). Pioneered in ACT (Action Chunking Transformer, 2023), adopted by most modern VLAs.

1. Full Model Pipeline

Pi0 is two transformers running in parallel, not one. The VLM backbone processes the scene and instruction; the action expert generates actions by cross-attending to the VLM's output.

Three things worth staring at:

  • The VLM runs once per action chunk (expensive — SigLIP + Gemma is most of the parameters). Its output feeds every action-expert block via cross-attention, and every Euler integration step.
  • The action expert is small (~300M in Pi0 base, vs ~3B for the VLM) and runs 10 times per chunk. Still cheap in aggregate because it doesn't include the vision/language encoders.
  • Training and inference are two different uses of the same model: training picks a random t and computes MSE on velocity; inference integrates the velocity field from t=0 to t=1 in small Euler steps.

2. Flow Matching vs Diffusion (DDPM)

We built a full DDPM diffusion model in the diffusion module. Flow matching is its simpler cousin. Same architecture, different training/sampling math.

DDPMFlow Matching (Pi0)
Path typeCurved (SDE)Straight line
Training targetε (noise)v = a - ε (velocity — constant along path)
Math neededNoise schedule α̅_t, variance logicLinear interp, velocity MSE
Sampling50–1000 steps (DDIM etc.)~10 Euler steps
ArchitectureSame transformerSame transformer

The math is beautifully simple. Sample t uniformly. Interpolate between noise and data. The path is straight, so the velocity is just (data − noise) — no schedule derivatives, no variance. Train to predict this velocity. Sample with Euler.

3. The Action Expert — Parallel Cross-Attention

The action expert is where Pi0 differs architecturally from prior VLAs. It's a separate, smaller transformer dedicated to the flow matching task, running parallel to the frozen VLM and cross-attending to it.

4. Code Walkthrough

compute_loss — flow matching training step ▼

Purpose: Single training step. Sample a random flow time t, interpolate noise and target, predict velocity, MSE loss. Same shape as DDPM training but simpler math.

def compute_loss(self, batch):
    images, token_ids = batch["images"], batch["token_ids"]
    proprio, action = batch["proprio"], batch["action"]
    B = action.size(0)

    # 1. Run VLM once to get context tokens
    vlm_tokens = self.encode_vlm(images, token_ids, proprio)

    # 2. Sample random flow time t and noise ε
    t = torch.rand(B, device=action.device)
    eps = torch.randn_like(action)

    # 3. Straight-line interpolation between noise and action
    t_exp = t.view(B, 1, 1)
    x_t = (1 - t_exp) * eps + t_exp * action

    # 4. True velocity is constant along the straight path
    v_target = action - eps

    # 5. Predict velocity with action expert (cross-attends to VLM)
    v_pred = self.action_expert(x_t, t, vlm_tokens)

    return F.mse_loss(v_pred, v_target)

Compared to DDPM training: no noise schedule, no α̅ lookup, no variance logic. v_target = a - ε is a one-liner. This is what people mean by "flow matching is simpler."

generate_action — Euler ODE inference ▼

Purpose: Integrate the learned velocity field from noise (t=0) to data (t=1) with N=10 Euler steps. The VLM is encoded once and reused.

@torch.no_grad()
def generate_action(self, images, token_ids, proprio, n_steps=10):
    B = images.size(0)

    # Expensive VLM pass — run ONCE, reuse across Euler steps
    vlm_tokens = self.encode_vlm(images, token_ids, proprio)

    # Start at pure noise
    x = torch.randn(B, self.horizon, self.action_dim, device=images.device)

    dt = 1.0 / n_steps
    for i in range(n_steps):
        t = torch.full((B,), i * dt, device=images.device)
        v = self.action_expert(x, t, vlm_tokens)        # cheap per step
        x = x + v * dt                                  # Euler step

    return x   # [B, H=50, action_dim] — the clean action chunk

Performance note: Caching vlm_tokens across Euler steps is the single most important optimization. Without it, you'd re-run the expensive VLM 10 times per chunk (10× slowdown). Any production Pi0 deployment does this.

ActionExpertBlock — one layer of the action expert ▼

Purpose: Three sub-layers per block: self-attention over action-chunk timesteps, cross-attention to VLM context tokens, FFN. The cross-attention is where VLA differs from a plain action policy — it reads the VLM's scene understanding as context.

class ActionExpertBlock(nn.Module):
    def __init__(self, d_action, d_vlm, n_heads=8):
        self.self_attn = nn.MultiheadAttention(d_action, n_heads, batch_first=True)
        # Cross-attn projects VLM keys/values from d_vlm → d_action
        self.ca_kv_proj = nn.Linear(d_vlm, d_action, bias=False)
        self.cross_attn = nn.MultiheadAttention(d_action, n_heads, batch_first=True)
        self.ffn = nn.Sequential(
            nn.Linear(d_action, 4 * d_action), nn.GELU(),
            nn.Linear(4 * d_action, d_action))
        # ... layer norms omitted for brevity

    def forward(self, action_tokens, vlm_tokens):
        # 1. Self-attention among action-chunk timesteps
        action_tokens = action_tokens + self.self_attn(
            action_tokens, action_tokens, action_tokens)[0]

        # 2. Cross-attention to VLM (read-only — VLM is frozen)
        kv = self.ca_kv_proj(vlm_tokens)
        action_tokens = action_tokens + self.cross_attn(
            action_tokens, kv, kv)[0]

        # 3. FFN
        action_tokens = action_tokens + self.ffn(action_tokens)
        return action_tokens

Subtle detail: the self-attention over the action chunk is NOT causal. All 50 timesteps attend to each other — we're co-generating them in a single forward pass, not rolling out autoregressively. This is a defining feature of action chunking.

6. Critical Optimizations

6.1 Cache VLM output across Euler steps ▼

The VLM is 80–90% of the parameters. Re-running it 10 times per Euler step would be an outrageous waste. Cache it once per chunk:

# Bad: re-encode VLM 10 times
for i in range(10):
    vlm_tokens = self.encode_vlm(images, text, proprio)   # WASTEFUL
    v = self.action_expert(x, t, vlm_tokens)
    x = x + v * dt

# Good: encode once, reuse
vlm_tokens = self.encode_vlm(images, text, proprio)       # once
for i in range(10):
    v = self.action_expert(x, t, vlm_tokens)              # cheap
    x = x + v * dt

Easily 10× speedup on the action-generation phase. Pi0 reference implementation does this by construction.

6.2 Freeze the VLM during fine-tuning ▼

During robot fine-tuning, the VLM (pretrained on web image-text) is typically frozen. Only the action expert and optional adapter layers train. Benefits:

  • Keeps generalist VLM capabilities (language understanding, novel object recognition)
  • Cuts training memory by ~10× (no VLM gradients or optimizer state)
  • VLM can be kept in INT8 for inference (since no gradients needed)
  • Faster convergence — the action expert only needs to learn the action distribution, not re-learn vision

Pi0 shows this works surprisingly well: the VLM's web-pretrained features are enough for robot manipulation, as long as the action expert is expressive enough.

6.3 INT8 VLM + FP16 action expert ▼
ComponentPrecisionWhy
Vision encoder (SigLIP)INT8Mature ViT quantization; biggest compute slice

Mixed INT8/FP16 gives ~2× speedup over all-FP16 with <1% success rate drop in Pi0 evaluations.

6.4 Reduce Euler steps (or distill) ▼

Pi0 uses 10 Euler steps. You can push this down with quality tradeoffs:

  • 5 steps: Small quality drop, 2× faster. Often acceptable for deployed policies.
  • 1 step: Need to distill. Pi0.5 and related papers train a student model to predict the full chunk in one shot.
  • Adaptive: Use more steps when uncertainty is high, fewer when confident.

Distillation is an active research area. The tradeoff is compute cost vs training effort — distillation takes extra training time but unlocks 10× deployment speedup.

6.5 Chunked re-prediction strategy ▼

Once you have a chunk of 50 actions, how often do you re-predict? Options:

  • Fixed rate: Re-predict every 25 timesteps (50% chunk overlap). Simple and robust.
  • Triggered: Re-predict when observed state deviates from predicted state by some threshold. Reactive to surprises.
  • Streaming: Predict the next chunk autoregressively (next chunk conditions on previous chunk). Enables longer-horizon plans.

Fixed rate with overlap is the production default. Latency is the critical constraint — if the VLA takes 200ms to predict but the chunk is 1000ms long, you have a 4× safety margin.

Humanoid compute is the frontier. Figure, 1X Neo, Tesla Optimus, Unitree H1 — all of these target Jetson Thor-class compute for their onboard inference. NVIDIA's Isaac GR00T project, Berkeley HumanPlus, and Stanford's TeleVision all explicitly target this hardware for VLA-style policies. Getting a 3B VLA to run at 5Hz on Thor is a milestone the field is actively working on.

Knobs for fitting into tighter compute budgets:

  1. Smaller VLM backbone (PaliGemma-400M instead of 2B)
  2. Fewer Euler steps (10 → 5)
  3. Smaller action expert (~100M — still enough for most tasks)
  4. INT8 VLM quantization (2× speedup)
  5. Reduce image resolution (224×224 → 168×168 saves ~1.8× vision FLOPs)

8. Connection to the Roadmap Ahead

VLA is a junction point. Several adjacent directions build on it:

  • Robot Foundation Models — next module. Multi-embodiment Pi0: train on arms, quadrupeds, and humanoids together with embodiment tokens. Same architecture, just different action spaces per embodiment.
  • World models — the flip side of VLA. Instead of p(action | observation), learn p(next_observation | action). Shares the same VLM backbone.
  • Occupancy networks and BEV (already covered) — perception backbones that feed VLA's vision side for multi-camera robot setups.
  • RL fine-tuning — the flow-matched action distribution can be PPO-fine-tuned. Treat the current VLA as the behavior policy, flow matching gives you gradient-free log-probability estimates.
  • Streaming VLA — receding-horizon chunks with continuous rollout. Active research area.
