Vision-Language Model (VLM)

LLaVA / Qwen-VL / Cosmos-Reason2 style: a ViT encoder wired to an LLM decoder through a tiny MLP projector. GPU inference study guide with shape traces, fusion mechanics, and edge deployment notes.

The Key Idea: VLM = ViT + Projector + LLM

A VLM is not a new architecture — it's three existing components wired together:

  1. A vision encoder (a ViT, with the [CLS] token and classification head removed) turns an image into a sequence of patch embeddings.
  2. A visual projector (a 2-layer MLP) maps those embeddings from the ViT's hidden dim into the LLM's hidden dim.
  3. An LLM decoder consumes image tokens and text tokens as one concatenated sequence and generates the next token autoregressively.

Image tokens form a prefix that the LLM attends to while generating text. In practice, chat templates sandwich them in control tokens: <s>...<im_start>[IMAGE TOKENS]<im_end>...<user>text<assistant>.

Why this works. You get to reuse battle-tested ViT and LLM checkpoints and only train the projector — that's LLaVA's stage-1 recipe: freeze ViT and LLM, train the 2-layer MLP on image-caption pairs. Optionally fine-tune in stage 2. Far cheaper than training a multimodal model from scratch.

1.2 Two phases of inference

Like any autoregressive LLM, a VLM's forward pass splits into two stages with very different cost profiles:

PhaseWhat happensRunsCompute profile
Vision encodeImage → patch features → projected image tokensOnce per imageDense batched GEMMs; compute-bound with Tensor Cores
PrefillFull forward pass on [image tokens + prompt], fills the KV cacheOnce per requestBig sequence (hundreds of tokens); mixed compute/memory
DecodeOne token at a time, attending to the cached K,VN_out times per requestTiny compute per step; memory-bound at batch 1

Two metrics come directly from this split:

  • TTFT (time-to-first-token) = vision encode + prefill. This is the wait the user feels before any output appears.
  • TPOT (time-per-output-token) = decode time. This is the steady-state streaming speed.

2. The Fusion Step

The non-obvious part of a VLM is how image and text modalities merge. Once you see it, it's trivially simple — which is why this pattern won.

2.1 Causal attention across modalities

Once concatenated, the sequence passes through causal self-attention. Two things to notice:

  • Image tokens see each other. Position 0 through N_v-1 can attend to any position 0 through N_v-1 — no causal restriction matters here because they're all in the past of any text token.
  • Text tokens see all image tokens plus all earlier text tokens. Text token at position N_v + i attends to positions 0 through N_v + i, so it has full access to the image and the conversation history up to that point.

The LLM doesn't have a separate mechanism for "paying attention to the image." Image tokens just happen to be early in the sequence, so every text-generation step naturally attends to them through the same causal self-attention that handles the text.

2.2 Position handling

Each token needs a position ID. Three common strategies:

StrategyUsed byIdea
1D flatLLaVAPositions 0..N_v+N_t-1 assigned in order. Image tokens look like arbitrary word tokens.
M-RoPEQwen2-VL3D RoPE for image tokens (temporal + height + width axes); text uses 1D.
InterleavedGemini, some ClaudeFine-grained inter-modal positioning; supports multiple images and video.

4. Image Token Reduction

A 448×448 image at patch size 14 gives 1024 visual tokens. That's before the user types a single word. With 3 images per prompt, you're at 3000 tokens before the conversation starts — and every token inflates the KV cache and prefill compute. Four common reduction strategies:

Pixel shuffle has won mainstream. It's the strategy used by Qwen2-VL, Qwen3-VL, and Cosmos-Reason2. The implementation is literally a reshape:

# Input: patch features from ViT, [B, H/P, W/P, d_v]
# H/P = W/P = 32 for 448/14 → 1024 patches in a 32×32 grid

# 2×2 pixel shuffle: merge adjacent patches into one bigger token
B, Gh, Gw, C = x.shape                      # [B, 32, 32, 1024]
x = x.reshape(B, Gh//2, 2, Gw//2, 2, C)     # [B, 16, 2, 16, 2, 1024]
x = x.permute(0, 1, 3, 2, 4, 5)               # [B, 16, 16, 2, 2, 1024]
x = x.reshape(B, (Gh//2)*(Gw//2), 4*C)      # [B, 256, 4096]
x = linear_down(x)                           # [B, 256, d_v] — downproject back

# 4× fewer tokens, same information (just reshuffled into the channel dim)

5. Code Walkthrough

The four core classes from vlm.py. Click any section to expand.

VisionEncoder — stripped-down ViT ▼

Purpose: Turn an image into a sequence of patch features for the projector. Same architecture as a classification ViT but with two parts removed: no [CLS] token (we want all patch tokens, not a single global vector) and no classification head (we're not predicting classes — we're producing features for the LLM).

class VisionEncoder(nn.Module):
    def __init__(self, img_size=448, patch_size=14, embed_dim=1024,
                 depth=24, num_heads=16):
        super().__init__()
        self.embed_dim = embed_dim
        self.patch_embed = PatchEmbed(img_size, patch_size, 3, embed_dim)
        num_patches = self.patch_embed.num_patches             # 1024 for 448/14

        # Pos embed has one slot per patch — no extra slot for CLS
        self.pos_embed = nn.Parameter(torch.zeros(1, num_patches, embed_dim))
        self.blocks = nn.ModuleList([
            VisionBlock(embed_dim, num_heads) for _ in range(depth)
        ])
        self.norm = nn.LayerNorm(embed_dim)

    def forward(self, images):
        x = self.patch_embed(images)          # [B, N_v, d_v]
        x = x + self.pos_embed
        for block in self.blocks:
            x = block(x)                      # bidirectional attn
        x = self.norm(x)
        return x                              # [B, N_v, d_v] — features

Deployment note: This module has a fixed output shape given a fixed input image size — no autoregression, no KV cache. It compiles cleanly to a static TensorRT engine and captures well with CUDA graphs. In production, you want this pre-compiled and warm at request time.

VisualProjector — the bridge MLP ▼

Purpose: Map vision features [B, N_v, d_v] into the LLM's embedding space [B, N_v, d_t]. This is the only module that's typically trained from scratch in LLaVA-style training — the ViT and LLM stay frozen while the projector learns to align the two spaces.

class VisualProjector(nn.Module):
    def __init__(self, vision_dim, text_dim):
        super().__init__()
        # Standard LLaVA-style: Linear → GELU → Linear
        self.fc1 = nn.Linear(vision_dim, text_dim)
        self.act = nn.GELU()
        self.fc2 = nn.Linear(text_dim, text_dim)

    def forward(self, x):
        # [B, N_v, d_v] → [B, N_v, d_t]
        return self.fc2(self.act(self.fc1(x)))

Why it's so small: Two linear layers and a GELU. Maybe 6M params for d_v=1024, d_t=2048. Less than 1% of the total VLM. Inference cost is negligible — fuses with the ViT's output projection in production.

Variants: Some models use a single Linear (cheaper, slightly worse). Flamingo uses a Perceiver resampler in place of this MLP. BLIP-2 uses a Q-Former. The LLaVA MLP style is the baseline everyone compares against.

LLMDecoder — causal transformer ▼

Purpose: Consume a sequence of embeddings (image tokens + text embeddings, pre-fused) and produce next-token logits. This is a standard decoder-only LLM — the multimodal magic already happened in the fusion step.

Note on the interface: This takes embeddings, not token IDs. The token-ID lookup happens outside, in the VLM wrapper, because we need to concatenate with image tokens before feeding the LLM.

class LLMDecoder(nn.Module):
    def __init__(self, vocab_size=32000, hidden_dim=2048,
                 depth=24, num_heads=16, max_seq_len=4096):
        super().__init__()
        self.hidden_dim = hidden_dim
        self.vocab_size = vocab_size

        # Token embedding table — used by outer VLM for text lookup
        self.embed_tokens = nn.Embedding(vocab_size, hidden_dim)
        self.pos_embed = nn.Parameter(torch.zeros(1, max_seq_len, hidden_dim))
        self.blocks = nn.ModuleList([
            LLMBlock(hidden_dim, num_heads) for _ in range(depth)
        ])
        self.norm = nn.LayerNorm(hidden_dim)
        self.lm_head = nn.Linear(hidden_dim, vocab_size, bias=False)

    def forward(self, embeds):
        # embeds: [B, N, hidden_dim] — already-fused image + text
        B, N, _ = embeds.shape
        x = embeds + self.pos_embed[:, :N]
        for block in self.blocks:
            x = block(x)                     # causal self-attention
        x = self.norm(x)
        return self.lm_head(x)                 # [B, N, vocab]

Real-world differences from this code: Modern LLMs use RoPE instead of learned position embeddings (better length generalization), SwiGLU instead of GELU in the MLP (slight quality boost), and GQA/MQA attention (fewer K,V heads than Q heads — cuts KV cache size 4–8×). All substitutable without changing the VLM structure.

VLM — the full model, the fusion step ▼

Purpose: Tie everything together. This class is the one that actually answers the question "what is a VLM?" — it shows exactly how image tokens get stitched into the text sequence before the LLM runs.

class VLM(nn.Module):
    def __init__(self, vision_encoder, llm):
        super().__init__()
        self.vision_encoder = vision_encoder
        self.projector = VisualProjector(
            vision_dim=vision_encoder.embed_dim,
            text_dim=llm.hidden_dim,
        )
        self.llm = llm

    def encode_image(self, images):
        # The step that gets CACHED in production —
        # deterministic, runs once per image.
        vision_feats = self.vision_encoder(images)    # [B, N_v, d_v]
        image_tokens = self.projector(vision_feats)   # [B, N_v, d_t]
        return image_tokens

    def forward(self, images, text_token_ids):
        # Vision path (once per image)
        image_tokens = self.encode_image(images)              # [B, N_v, d_t]

        # Text path: token IDs → embeddings via LLM's table
        text_embeds = self.llm.embed_tokens(text_token_ids)   # [B, N_t, d_t]

        # *** THE FUSION STEP ***
        # Both tensors are in the LLM's embedding space now —
        # concatenating them is well-defined.
        combined = torch.cat([image_tokens, text_embeds], dim=1)

        # LLM sees one sequence, doesn't distinguish modalities
        return self.llm(combined)                             # [B, N_v+N_t, vocab]

What's NOT shown here: The autoregressive generation loop (KV cache, sampling) and the real chat template that surrounds image tokens with control markers like <im_start> / <im_end>. Those live in the inference stack (vLLM, TRT Edge-LLM, llama.cpp), not in the model architecture itself.

7. Critical Optimizations for Inference

7.1 FlashAttention — mandatory for long prefill ▼

Vanilla attention materializes the [B, H, N, N] score matrix in HBM. For a VLM with N=1024 visual tokens and 16 heads at batch 1, FP16, that's 16 × 1024 × 1024 × 2 = 32 MB per layer — written then read back through HBM. Across 28 LLM layers: ~900 MB of traffic just for the attention matrix.

FlashAttention tiles the computation in SRAM and never materializes the full score matrix. On Jetson with ~102 GB/s LPDDR5, cutting 900 MB of HBM traffic is directly TTFT. On-by-default in modern frameworks (PyTorch 2.0+ SDPA, vLLM, TensorRT).

import torch.nn.functional as F
# PyTorch picks FlashAttention automatically on supported GPUs
x = F.scaled_dot_product_attention(q, k, v, is_causal=True)
7.2 Vision encoder caching ▼

The vision encoder is deterministic: same image always produces the same token sequence. In multi-turn chat, or when the same image appears across many queries (dashcam frames, surveillance feeds), cache the post-projector tokens. One full ViT forward (60 GFLOPs on VLM-Base) saved per cache hit.

# Pseudo-code: hash image → cached image_tokens
def encode_image_cached(image):
    key = sha256(image.tobytes())
    if key not in CACHE:
        CACHE[key] = self.encode_image(image.unsqueeze(0))
    return CACHE[key]

vLLM supports this natively since v0.5+. TensorRT-LLM requires manual wiring but is straightforward.

7.3 Separate vision and LLM engines ▼

Production deployments split the vision encoder and LLM decoder into two independent inference engines. Why:

  • Vision runs once per image; LLM runs once per output token. Different cost profiles want different batch strategies.
  • Vision can batch across requests (static shape); LLM needs continuous batching (dynamic shape).
  • Engines can live on different devices in multi-GPU deployments.
  • Quantization differs — ViT tolerates INT8 well, LLM decoder usually wants INT4 weights + FP16 activations.
7.4 Chunked prefill — hide image cost behind decode ▼

Standard prefill processes the full [N_v + N_t] input in one kernel launch, blocking any other request. Chunked prefill breaks it into smaller pieces (say, 256 tokens at a time) and interleaves with decode steps from other requests.

For a serving system with 10 concurrent requests, this prevents any single VLM request with a big image from monopolizing the GPU for 200+ ms. Instead, the GPU alternates between "prefill chunk for request A" and "decode step for requests B-J."

Supported by vLLM (on by default for chunked inputs) and TensorRT-LLM (use_chunked_context). Directly relevant if you're serving multiple users.

7.5 Quantization — different for each component ▼

Different VLM components have different quantization tolerances:

ComponentTypical quantizationWhy
Vision encoderINT8 or FP16ViT tolerates INT8 well; static shape makes PTQ easy
ProjectorFP16Tiny, not worth quantizing; accuracy-sensitive
LLM decoder weightsINT4 (AWQ or GPTQ)Biggest component; memory-bound at decode
LLM decoder activationsFP16W4A16 is the sweet spot for edge deployment
KV cacheFP16 or INT8INT8 KV saves half the memory with negligible quality loss

On my Orin Nano runs, W4A16 AWQ + FP16 vision + FP16 KV fit the 2B model in 4.3 GB with room to spare.

From my three-runtime benchmark of Cosmos-Reason2-2B on the 8 GB Orin Nano Super (JetPack 6.2.2, MAXN_SUPER power mode):

Three observations that came out of the data:

  1. The image TTFT gap IS the visual-token prefill cost. Every runtime's image TTFT is 2–13× its text TTFT. The gap is time spent doing the full vision encode plus attention over 256–1024 visual tokens. This is the prefill-dominated story from §3, visible in real numbers.
  2. Decode TPS tracks LPDDR5 bandwidth, not compute. 38–60 TPS across three very different runtimes, all on the same 102 GB/s memory subsystem. Decode is memory-bound at batch 1; bandwidth is the real bottleneck, not TOPS.
  3. TRT Edge-LLM wins peak memory by a lot. 4.3 GB vs 6.9 GB — 37% less than vLLM. Why: fully-fused engine with predictable static allocations, no Python overhead, INT4 weights pre-packed. Cost: 10-minute engine build + two workarounds (CMA kernel param + ONNX graph rewrite).
Production trade-off made concrete. llama.cpp: portable generic CUDA kernels, small-chunk memory allocation — no build step, works everywhere, middling speed. TRT Edge-LLM: device-specific tactic selection, large contiguous scratch buffers — fastest when built, failed the first time because Myelin's 1 GiB allocation hit NvMap fragmentation on JetPack 6.2.2. The portability/optimality trade-off is not just a talking point at the edge.
